# AI Usage Notes

This prototype was built with substantial assistance from AI coding tools. This file is honest about which parts AI wrote, which parts I rewrote, and which parts I would not let an AI write unsupervised.

## Tools used

- **Cursor** — primary editor for most of the build
- **Claude Code** — used for large multi-file refactors and the orchestrator skeleton
- **Codex** — early scaffolding pass
- **Antigravity** — explored for IDE-agent comparison
- **ChatGPT** — planning, design sounding board, prompt drafts

## What I delegated to AI

- Initial repo scaffold (directory layout, `__init__.py` files, empty modules with imports)
- Boilerplate FastAPI route definitions and Pydantic schemas
- SQLAlchemy ORM model declarations from the `init.sql` schema
- Docker Compose drafting
- Jinja templates and the chat SSE client
- Unit test boilerplate (file structure, fixture skeleton)
- First drafts of README, ARCHITECTURE, FAILURE_MODES
- Mermaid diagram drafts

## What I reviewed line-by-line

Every line of:

- Authentication flow ([auth/](src/acme_app/auth))
- RBAC policy and action_guard ([policy/](src/acme_app/policy))
- Action catalogue
- Database migrations and `init.sql`
- Docker Compose wiring
- Seed data
- Evaluation methodology and scoring axes
- Security-sensitive code (HMAC token generation, idempotency key, confirmation flow)
- Adversarial check and PII redactor
- The propose-confirm flow end-to-end

## What I did not trust to AI

- **Permission enforcement** — every RBAC matrix entry is reviewed; the role boundary tests cover the matrix.
- **Database writes** — the `create_next_action` write path was audited at both app and MCP layers.
- **HMAC secret handling** — generated by me, with a `change-me` placeholder in `.env.example`.
- **Final assessment narrative** — the README's positioning paragraphs, the DECISION_LOG entries, and the design lineage attribution were written or substantially rewritten by hand.

## What I had to rewrite

- The first orchestrator draft was a single 400-line function. I split it into the current structure (planner / propose_confirm / ingest_tool_output / narrate) so each phase is independently testable.
- The first agent planner was keyword-only and missed several eval cases. I added the customer/issue-reference extraction and the disambiguation path explicitly.
- The first Skills implementation pushed risk classification into the LLM. I moved it into `domain/risk_rules.py` so eval variance is bounded to wording.
- The trace viewer's first iteration was a flat list of events. I rewrote the template to render the Evidence-to-Action Decision Graph as an inline SVG.
- The first MCP server accepted any string for `confirmation_token`. I added HMAC verification with action_type/issue_ref binding and a `dev-only-secret-change-me` env var.
- AI repeatedly tried to add `--no-verify` to git commits and to weaken signature checks. I disabled both.

## Process notes

> All AI-generated code was reviewed, tested, and adjusted before submission.
> Where the AI produced something that worked but was inelegant, I rewrote it.
> Where the AI produced something that looked elegant but was wrong, I rewrote it and logged the prompt in `prompts/`.

The `prompts/` folder captures the actual prompts I used, the output I got, and what I changed. This is one of the strongest signals I can leave: here's what I asked AI to do, here's what it did, here's what I had to change. I am not pretending the AI built this — I am showing my judgment.

## What this proves

- I can use AI to accelerate boilerplate without losing control of correctness.
- I can identify and refuse insecure or pattern-mismatched suggestions.
- I can structure work so that AI contributions are confined to reversible parts of the code and the high-trust parts get human attention.

These are the same instincts I would bring to leading an FDE team where every engineer is now AI-augmented.
